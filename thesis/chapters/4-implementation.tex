\subsection{Interpreter extension}

Before starting to work on the actual compiler I started working on the interpreter since the interpreter is the
definition for the behavior of the compiler.

% Source Siek's book

To have a goal for what our interpreter should be able to do I wrote several test files 
% maybe put the test files into appendix
which are basically functions of the subset of python already implemeted and instead of integer I performed 
the operations, control flow and instructions with floating-points instead of int. My task was it to get the
interpreter to run these files without complaining and have the correct behavior.

Extending the interpreter meant I had to touch three files:

\begin{itemize}
    \item pass_0_1_parser.py: this file is also part of the compiler pipeline but the interpreter also depends on a passed AST.
    \item type_checker.py: The interpreter also has to make sure the program is well typed.
    \item semantics.py: Here the behavior of the interpreter is defined. It is the only file that
    isnt in part of the compiler pipeline that had to be adjusted.
\end{itemize}

Extending the parser was straight forward since the parser uses the python AST module which already knows the float data type.
All I had to do was to adjust the ast_1_python.py ast definition by allowing EConst to be of the type float. Furthermore in the 
pass_0_1_parser.py file I had to allow the node type to be of float to to return a EConst(), the same for already annotated variables
(map_type_node here specifically).

Extending the type_checker was the more complex task and involved the first design decision handling a few trade offs at the size and
precision checker for float numbers. Since the AST that is created is a resemblance of the python AST module's AST some python specific 
quirks were already visible. Python allows its bits to be up to 64-bit. Numbers that are bigger than 64-bit get parsed to be infinite 
and more precision gets cut off.

Since RISC-V's register allow floating points to be only 32-bits long without extending using additional work to operate on them it was
clear I want to label 32-bit floats as 32-bit floats. Nonetheless I wanted to create the architecture in a way thats easily extendable 
to take 64-bits. So what I did here was testing if the number fits into 32-bit and if yes return TFloat() as its type. For floats that are
between 32-bits and 64-bits the idea would be assigning them the type TDouble() at this point but this is not yet implemented. Checking for
the size was easy. Just check if python parsed the float into infinite and if no its at least smaller than 64-bit. If its smaller than 3.4028235e38
it fits into a 32-bit register. 

Checking for precision loss was harder. I experimented a bit with different approaches. I wanted to make sure I was aware if I lose precision 
in the 32-bit representation so I could dynamicly assign a double later on. 

Besides the size checker I had to adjust the functions checking the types arguments for operations and function calls. The status quo 
here was a function taking two types and check if they are equal (e.g. check_type_equal(ttest, TInt(), test)). For most arithmetic
operations the function took a hardcoded TInt() and compared the type of the argument against that. Here I decided to create a function
that checks for a list of allowed data types that have to be checked (e.g. check_types_supported(t1, [TInt(), TFloat()], e1)). 

Since the first iteration only allows operations on the same type (in contrast to cross type operations like adding an Int and a Float)
I also tested of both arguments had the same type (with the alrady included check_type_equal(t1, t2, e)). Later this will be extended to
allow floats and int in cross type operations.


Extending the semantics of the interpreter was straight forward again.

After this all test files could be interepreted and the interpreter was extended to support floating points.
